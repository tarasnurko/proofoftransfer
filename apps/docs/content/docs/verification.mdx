## Overview

Third-party verification confirms that a prover's ZK proof corresponds to real on-chain transfers. The verifier independently fetches/uploads transfers, builds a merkle tree client-side, and the server compares roots + runs the ZK verifier.

## Flow

```
Verifier (browser)                          Server
─────────────────                          ──────
1. Connect wallet
2. Fetch transfers (API) or upload CSV
3. Sign claim (EIP-712) → derive nullifier
4. Client-side self-verify check
   (derivedNullifier === proof.nullifier → reject)
5. Build merkle root from transfers
   (Barretenberg WASM, poseidon2)
6. Send { proofId, nullifier, merkleRoot } ──→  7. Validate inputs
                                                8. Check not self-verify (nullifier match)
                                                9. Check not already verified successfully
                                               10. Compare merkle roots (verifier vs claim)
                                               11. ZK verify (UltraHonk backend)
                                               12. Delete old failed verifications for this nullifier
                                               13. Create verification record (pass/fail)
                                               14. Return { isValid, error, stats }
                                           ←──
15. Update verification stats from server response
16. Show result:
    - Success → toast + hide verify form ("Proof has been verified successfully")
    - Failure → toast + persistent error banner in verify card
```

## Identity Model

- No wallet addresses stored in DB
- Identity = nullifier = `poseidon2(r, s)` of EIP-712 signature
- Deterministic per (wallet, claim) pair
- Self-verification detected by comparing derived nullifier with proof's nullifier

## Merkle Tree

- Height: 20 levels
- Hash function: poseidon2
- Leaves: `poseidon2(from, to, contractAddress, value, timeStamp, txHash)`
- Transfers sorted by `timeStamp` ascending before hashing
- Built client-side using Barretenberg WASM (`@aztec/bb.js`)
- Server only compares the root against the claim's stored root

## Verification Counting Rules

Each user (identified by nullifier) gets at most ONE verification record per proof at a time.

| Scenario | Action | Effect |
|----------|--------|--------|
| First failure | Create failed record | `failed += 1` |
| Retry failure (same user) | Delete old failure, create new | Count unchanged |
| Success after failure | Delete old failure, create success | `failed -= 1, successful += 1` |
| First success (no prior fail) | Create success record | `successful += 1` |
| Already succeeded | Blocked | Cannot re-verify |

This is implemented via:
- `deleteFailedVerificationsByNullifier` — removes previous failed record before each attempt
- `createVerification` — inserts new record (pass or fail)
- `getSuccessfulVerificationByNullifier` — blocks re-verification after success

After each verification attempt, the server returns updated stats from `getVerificationStats` so the UI always reflects the true count.

## UI States

The verify card shows different states:

| State | What's shown |
|-------|-------------|
| Not connected | "Connect your wallet" button |
| Connected, no transfers | Fetch/upload transfers UI, verify button disabled |
| Connected, has transfers | Verify button enabled |
| Verification failed | Persistent error banner (red, with error message) + verify button still available for retry |
| Verification succeeded | Form hidden, "Proof has been verified successfully" message with checkmark |
| Already verified (page load) | Same as succeeded — form hidden |

The error banner persists across the session until the user successfully verifies (clears the error) or reloads the page. Stats (`X verified`, `Y failed`) are shown in the card header and update in real-time from server responses.
